Time & Space Complexity

Master algorithm analysis and understand how code efficiency scales with input size

📖
Understanding Through Story
🖊️ The Lost Pen Problem

Imagine a classroom of 100 students where you gave your pen to someone. Now you need to find that pen without knowing who has it. Here are different approaches:


O(n²) — The Inefficient Way: You ask the first person if they have the pen, and also ask them about ALL other 99 students. Then you do the same with the second person, and so on.


O(n) — The Linear Way: You simply go to each student one by one and ask "Do you have my pen?" This takes 100 questions maximum.


O(log n) — The Smart Way: You divide the class into two halves and ask "Is the pen on the left side or right side?" Keep dividing until you find it. Only ~7 questions!

Key Insight

  • Time complexity measures how the number of operations grows as input size increases
  • It's NOT about actual execution time (which depends on hardware)
  • It's about counting how many times statements execute
  • Different algorithms solve the same problem with vastly different efficiency
⏱️
What is Time Complexity?

Time complexity measures how the runtime of an algorithm grows as the input size increases. It tells us the rate of growth, not the exact time.

⚡ Why Not Measure Actual Time?

Consider finding the maximum number in an array:

• N = 10: Your computer might take 0.5 ms
• N = 10,000: Your computer might take 0.2 ms

Wait — same code, but larger input is faster? 🤔

The reason: Actual time depends on machine speed, CPU load, memory, compiler optimizations, and many other factors. This makes actual time unreliable for comparing algorithms!

Instead, we count how many times each statement executes relative to input size. This gives us a machine-independent measure of efficiency.

Key Takeaways

  • O(1) and O(log n) scale incredibly well — even with millions of items
  • O(n) and O(n log n) are acceptable for most real-world applications
  • O(n²) becomes problematic with large inputs (avoid when possible)
  • O(2ⁿ) is practically unusable for n > 30

Asymptotic Notations: Big-O, Omega, Theta
Big-O (O)
Upper Bound — Worst Case

Represents the maximum time an algorithm can take. Guarantees the algorithm won't be slower than this.

If f(n) ≤ c·g(n) for all n ≥ n₀
then f(n) = O(g(n))
Omega (Ω)
Lower Bound — Best Case

Represents the minimum time an algorithm needs. Guarantees the algorithm won't be faster than this.

If f(n) ≥ c·g(n) for all n ≥ n₀
then f(n) = Ω(g(n))
Theta (Θ)
Tight Bound — Average Case

Represents both upper and lower bounds. The algorithm's time is sandwiched between these bounds.

If c₁·g(n) ≤ f(n) ≤ c₂·g(n)
then f(n) = Θ(g(n))
🎯 When to Use Each Notation

Big-O (O): Most commonly used. Perfect when you want to guarantee an algorithm won't exceed a certain time. Example: "Bubble Sort is O(n²)" means it will never be worse than quadratic.


Omega (Ω): Useful for proving a problem requires at least a certain amount of time. Example: "Comparison-based sorting is Ω(n log n)" means you can't do better than this.


Theta (Θ): Most precise. Used when upper and lower bounds match. Example: "Merge Sort is Θ(n log n)" means it's always n log n, in all cases.

Space Complexity

Space complexity measures how much memory an algorithm uses as input size grows.

Components of Space Complexity

  • Input Space: Memory used to store the input data
  • Auxiliary Space: Extra/temporary space used by the algorithm
  • Total Space: Input Space + Auxiliary Space

Key Takeaways

Time Complexity

  • Measures how runtime scales with input size
  • Focus on rate of growth, not exact time
  • Drop constants and lower-order terms: 3n² + 5n + 10 → O(n²)
  • Big-O gives worst-case guarantee
  • Prefer O(1), O(log n), O(n) when possible

Space Complexity

  • Measures memory usage as input grows
  • Includes auxiliary space + input space
  • Recursive calls count toward space complexity
  • In-place algorithms use O(1) auxiliary space
  • Sometimes you trade time for space (or vice versa)
