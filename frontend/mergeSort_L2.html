BASE CASE: Single element
if left >= right: return
DIVIDE: Find midpoint

RECURSE LEFT: Sort left half
mergeSort(arr, left, mid)
RECURSE RIGHT: Sort right half
mergeSort(arr, mid+1, right)
MERGE: Combine both halves
merge(arr, left, mid, right)

MID & RIGHT: Compute bounds
mid = min(left+width−1, n−1)

MERGE: Merge [left..mid] & [mid+1..right]

Try Different Scenarios

Best Case — Already Sorted

Even when already sorted, Merge Sort always does O(N log N) comparisons — it still divides and merges every subarray. No early-exit like Bubble Sort. But comparisons are minimized: each merge only needs N/2 comparisons since left elements are always ≤ right.

🗄 Array: [3, 7, 12, 18, 25, 31] — sorted, fewest comparisons per merge

Average Case — Random Order

Random array — a typical number of comparisons and copies occur during each merge. Watch the divide phase break the array into halves, and the merge phase reconstruct it in sorted order.

Worst Case — Reverse Sorted

Reverse-sorted array forces the maximum number of comparisons in each merge — every element from left and right halves interleaves. Still O(N log N), but with the highest constant.

🗄 Array: [90, 70, 50, 30, 20, 10] — reverse sorted, max comparisons
