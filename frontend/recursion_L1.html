Understanding Recursion

💡 What is Recursion?

The process in which a function calls itself directly or indirectly is called recursion, and the corresponding function is called a recursive function. A recursive algorithm takes one step toward the solution and then recursively calls itself to continue. The algorithm stops once we reach the solution. Since a called function may further call itself, it is essential to provide a base case to terminate the recursion process.

🤔 Why Use Recursion?

  • Recursion helps in logic building — recursive thinking helps break complex problems into smaller subproblems
  • Recursive solutions form the basis for Dynamic Programming and Divide and Conquer algorithms
  • Certain problems are elegantly solved using recursion — Tower of Hanoi, tree traversals (Inorder/Preorder/Postorder), DFS of Graph
  • Recursive code is often cleaner and shorter than an equivalent iterative version

🔄 How Does Recursion Work?

Every recursive function has two key parts: a base case that stops the recursion, and a recursive case that breaks the problem into a smaller version and calls itself. Each call creates a new stack frame with its own local variables. When the base case is reached, the stack unwinds — each call returns its value to the caller, eventually solving the original problem.

Key Concepts

Base Case
The stopping condition for recursion. Without a correct base case, recursion runs forever and causes a stack overflow.
Recursive Case
The part where the function calls itself with a smaller/simpler input, progressing toward the base case.
Call Stack
Each recursive call creates a new stack frame. The stack follows LIFO — the last call resolves first. Depth = O(n) for linear recursion.
Winding vs Unwinding
Winding = building up recursive calls. Unwinding = resolving them as the stack pops. Code before recursive call runs on winding; after on unwinding.
Memoization
Caching results of recursive calls to avoid redundant computation. Converts Fibonacci from O(2ⁿ) to O(n). Foundation of dynamic programming.
Tail Call Optimization
When recursive call is the last operation, some compilers reuse the same stack frame — reducing space from O(n) to O(1).
🧠

Memory Allocation in Recursion

  • Whenever a function is called from main(), memory is allocated to it on the stack
  • A recursive function calls itself — each call creates memory on top of the previous frame, with a different copy of local variables
  • The internal system uses a stack (LIFO) because the last called function finishes first
  • When the base case is reached, the function returns its value — memory is de-allocated and the process continues upward
  • Infinite recursion leads to Stack Overflow when the call stack runs out of memory
📝

Steps to Implement Recursion

Every recursive solution follows the same 4-step framework:

1
Define the Base Case
Identify the simplest case for which the solution is known or trivial. This is the stopping condition — it prevents infinite recursion.
2
Define the Recursive Case
Express the problem in terms of smaller subproblems. Break it down and call the function recursively with a smaller input.
3
Ensure Termination
Make sure the recursive function eventually reaches the base case. Each call must move closer to the base case — never away from it.
4
Combine the Solutions
As the stack unwinds, combine results of subproblems to build the solution to the original problem.

Classic Recursive Examples

Three essential examples that demonstrate the power and structure of recursion.

➕ Example 1: Sum of Natural Numbers

Compute the sum of first n natural numbers recursively. sum(3) = 3 + sum(2) = 3 + 2 + sum(1) = 6

Problem:

Input: n = 3 → Output: 6 (1+2+3)
Input: n = 7 → Output: 28 (1+2+3+4+5+6+7)

  • Base Case: At n == 1, return 1 (stop recursion)
  • Recursive Case: sum(n) = n + sum(n-1)
  • Execution flow (n=3): sum(3) → sum(2) → sum(1)=1 → sum(2)=3 → sum(3)=6

int sum(int n) {

🔢 Example 2: Factorial of a Number

The factorial of n (n ≥ 0) is the product of all positive integers from 1 to n. fact(5) = 5 × 4 × 3 × 2 × 1 = 120

  • Base Case: fact(0) = 1 (by mathematical definition)
  • Recursive Case: fact(n) = n × fact(n-1)
  • Execution (n=5): 5×4×3×2×1×1 = 120

🌀 Example 3: Fibonacci Series

Each Fibonacci number is the sum of the two preceding ones. fib(n) = fib(n-1) + fib(n-2). Series: 0, 1, 1, 2, 3, 5, 8, 13...

  • Base Cases: fib(0) = 0, fib(1) = 1 (or fib(2) = 1)
  • Recursive Case: fib(n) = fib(n-1) + fib(n-2)
  • Recurrence: T(n) = T(n-1) + T(n-2) + O(1) → Exponential O(2ⁿ) — can be optimized with memoization

Types of Recursion

Understanding the different forms of recursion helps in identifying the right approach for a problem.

🔄 Direct Recursion
A function calls itself directly within its own body. The most common form of recursion.

void fun() { fun(); }
🔁 Indirect Recursion
Function A calls Function B, and Function B calls Function A — forming a cycle of calls.

void A() { B(); } void B() { A(); }
🐾 Tail Recursion
The recursive call is the very last statement in the function. No work is done after the return. Can be optimized by the compiler to use O(1) stack space.
🌳 Tree Recursion
A function makes more than one recursive call per invocation. Fibonacci is a classic example. Leads to exponential time without memoization.

Stack Overflow in Recursion

🚨 When Does Stack Overflow Occur?

If the base case is not reached or not defined correctly, the stack overflow problem may arise. Each recursive call adds a new frame to the stack. When the system's memory is exhausted due to unending function calls, a Stack Overflow error occurs.

⚠
Wrong Base Case → Stack Overflow Example
▼

The following code has a wrong base case — it checks n == 100 but we call fact(10), so n will never reach 100 during downward recursion. This creates infinite recursion and eventually causes a stack overflow.

Stack Overflow Example (C++)
// ❌ WRONG — This causes stack overflow for fact(10)
int fact(int n) {
    if (n == 100)        // Wrong base case! n=10 will never reach 100
        return 1;
    else
        return n * fact(n - 1);  // Infinite descent: 10→9→8→...→-∞
}

// ✅ CORRECT — Base case n==0 is always reachable
int factCorrect(int n) {
    if (n == 0)             // ✅ n will always eventually reach 0
        return 1;
    return n * factCorrect(n - 1);
}

Recursion vs Iteration

Every recursive program can be written iteratively, and vice versa. Choosing between them depends on readability, performance, and problem structure.

Aspect Recursion Iteration
Code Clarity Often cleaner and more expressive for tree/graph problems Can be verbose but straightforward for loops
Space O(n) call stack space — can cause stack overflow O(1) extra space (no call stack overhead)
Time May have overhead from function call setup/teardown Generally faster due to no function call overhead
Infinite loop risk Stack overflow if base case missing or wrong Infinite loop if termination condition is wrong
Best for Tree traversal, DFS, D&C, Tower of Hanoi, Fibonacci Simple array operations, counting, summing
✅ Advantages of Recursion
Clean and simple code. Naturally expresses problems like tree traversal, DFS, and divide-and-conquer. Forms basis for DP and backtracking.
⚠️ Disadvantages of Recursion
Higher space complexity due to call stack. Can be harder to debug across multiple levels. Risk of stack overflow with deep recursion.

Common Applications of Recursion

Recursion is used in almost every advanced algorithmic technique:

1
Tree & Graph Traversal
Inorder, Preorder, Postorder tree traversal. DFS of graphs. All naturally expressed with recursion.
2
Sorting Algorithms
Merge Sort and Quick Sort divide arrays into subarrays, sort them recursively, then merge/combine.
3
Divide & Conquer
Binary Search, Strassen's Matrix Multiplication — break problems into halves and solve recursively.
4
Backtracking
N-Queens, Sudoku Solver, Rat in a Maze — explore all paths, backtrack when stuck, recurse forward.
5
Tower of Hanoi
Classic recursive puzzle — move n disks using 3 pegs. Impossible to solve elegantly without recursion.
6
Dynamic Programming
Memoized recursion is the basis of DP. Top-down DP = recursion + caching of subproblem results.
