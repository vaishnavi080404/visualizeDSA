BST — Best Case O(1): target is the root

When what you're looking for is the root node, BST finds it in 1 step. Click "Search Root" to see it — then "Search Leaf" to see O(log n).

📋 Log

Binary Tree — Best Case: Perfect level-order build

Inserting into a Binary Tree always fills left-to-right, level-by-level — producing the most balanced (minimum height) shape possible.

📋 Log

N-ary Tree — Best Case: Wide and shallow

A root with many children keeps height = 1. All nodes are just 1 hop from root. The more children per node, the shorter the tree.

📋 Log

BST — Average Case O(log n): half eliminated per step

Each comparison eliminates half the remaining tree. 7 nodes → 3 steps. 1 million nodes → 20 steps. Step through to see it.

📋 Log

Binary Tree — Average Case O(n): Inorder Traversal

Inorder traversal visits every node exactly once — Left → Root → Right. Total cost is always O(n) regardless of tree shape.

📋 Log

N-ary Tree — Average Case O(n): BFS Level-Order Traversal

BFS visits nodes level by level, using a queue. Every node is visited exactly once — total O(n) work.

📋 Log

BST — Worst Case O(n): sorted insertion (skewed tree)

Inserting 10→20→30→40→50→60 creates a straight-line chain — every search must traverse all N nodes. This is why real databases use self-balancing trees.

📋 Log

Binary Tree — Worst Case O(n): linear scan (no ordering)

Binary Trees have no ordering rule — searching for a value means checking every node one by one. No shortcut is possible.

📋 Log

N-ary Tree — Worst Case O(n): chain tree (branching = 1)

When every node has exactly 1 child, the N-ary tree becomes a linked list. Height = n, search must visit all nodes.

📋 Log
