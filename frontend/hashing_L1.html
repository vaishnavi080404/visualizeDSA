Understanding Hashing

💡 What is Hashing?

Hashing refers to the process of generating a small, fixed-sized output (used as an index in a table) from an input that is typically large and variable in size. It uses mathematical formulas known as hash functions to perform this transformation. The result determines an index or location for storing an item in a data structure called a Hash Table.

🤔 Why Use Hashing?

Hashing is one of the most widely used data structures after arrays. It mainly supports search, insert, and delete in O(1) average time, which is more efficient than arrays, linked lists, and self-balancing BSTs. Common uses include dictionaries, frequency counting, and maintaining data for quick access by key.

  • Database Indexing — fast record lookup without scanning all rows
  • Cryptography — password hashing, digital signatures, checksums
  • Caches — O(1) cache hit/miss checks in browser or CPU caches
  • Symbol Tables — compilers store variable names in hash maps
  • Dictionaries — Python dict, JavaScript Map, C++ unordered_map

🔄 How Does Hashing Work?

A key is passed to a hash function, which computes an integer index. That index is used to store or retrieve the value in a hash table (typically an array of lists). The goal is to distribute keys as uniformly as possible across the table, minimising collisions.

How Hashing Works — Step by Step

Let's walk through a concrete example: storing the string set {"ab", "cd", "efg"} in a hash table of size 7.

1
Assign Character Values
Map every letter to a number. 'a'=1, 'b'=2, 'c'=3, 'd'=4, 'e'=5, 'f'=6, 'g'=7 …
a=1, b=2, c=3, d=4 …
2
Compute Key Sums
Sum the character values for each string to get a numeric representation of the key.
"ab" = 1+2 = 3
"cd" = 3+4 = 7
"efg" = 5+6+7 = 18
3
Apply Hash Function
Take sum mod table_size (7) to get the final index. This maps any key into range [0, 6].
h(k) = sum(k) mod 7
4
Compute Indices
Apply the formula to each key to find its destination slot in the hash table.
3 mod 7 = 3 ✓
7 mod 7 = 0 ✓
18 mod 7 = 4 ✓
5
Store in Table
Place each key (and its value) at the computed index. Now any key can be found in O(1) by recomputing its hash.
table[0] = "cd"
table[3] = "ab"
table[4] = "efg"
6
Handle Collisions
If two keys hash to the same index, use chaining (linked list at that slot) or open addressing (probe for next empty slot).

Key Concepts

🔤 Key
The input fed to the hash function. Can be any data type — string, integer, object. Must be hashable (immutable in Python). Example: username, product ID.
ƒ Hash Function
A mathematical formula that converts a key into an integer index. Must be deterministic, fast to compute, and distribute keys uniformly across the table.
📋 Hash Table
The underlying array-of-buckets data structure. Each bucket holds one key-value pair (open addressing) or a list of pairs (chaining). Typical size is a prime number.
💥 Collision
When two different keys produce the same hash index. Unavoidable by the pigeonhole principle. Handled via chaining, linear probing, or quadratic probing.
⚖️ Load Factor
Ratio = n / capacity (n = entries stored). Higher load factor → more collisions. Most implementations rehash when load factor exceeds 0.7–0.8.
🔄 Rehashing
When the load factor is too high, a new larger table is allocated (usually 2× capacity) and all existing keys are re-inserted into it. O(N) one-time cost.
🧩

Components of Hashing

There are three major building blocks of any hashing system:

1️⃣ Key

A key can be any string or integer that is fed as input to the hash function. The key identifies what you want to store or retrieve. Keys must be unique within the hash table (like a primary key in databases). In most languages, keys must be immutable (strings, numbers, tuples — not lists).

2️⃣ Hash Function

Receives the input key and returns an integer index (the hash index) into the underlying array. A good hash function should:

  • Be deterministic — same key always returns same hash
  • Be fast — O(1) or O(key_length) computation
  • Produce uniform distribution — minimize clustering
  • Minimize collisions — rare but unavoidable
Common hash functions: Division method h(k) = k mod m, Multiplication method, Universal hashing, MurmurHash, FNV hash, SHA-256 (cryptographic).

3️⃣ Hash Table

Hash table is typically an array of buckets (lists or slots). It stores values corresponding to the keys. Each data value has its own unique index computed by the hash function. The table size is usually chosen to be a prime number to reduce clustering.

🗂️

Two Forms of Hashing

Hash Set — Collection of Unique Keys

Stores only keys (no values). Used to check membership, deduplicate, or find unique elements quickly.

Python: set JS: Set C++: unordered_set Java: HashSet

Hash Map — Key-Value Pairs with Unique Keys

Stores (key → value) pairs. The most commonly used form — like a dictionary where you look up a word (key) to get its definition (value).

When NOT to Use Hashing

Hashing is powerful but not a universal tool. There are cases where other data structures are more appropriate:

🌳 Need Sorted Data
When you need to maintain sorted order along with search/insert/delete, use a Self-Balancing BST (AVL Tree, Red-Black Tree) — O(log N) for all ops but sorted.
🔠 Prefix Search on Strings
When keys are strings and you need prefix search (autocomplete), use a Trie. Hash tables can't efficiently handle prefix queries.
🏔️ Floor / Ceiling Queries
When you need to find the greatest key ≤ x or smallest key ≥ x, use a Self-Balancing BST or Sorted Array. Hash tables are unordered.
📊 Range Queries
When you need all keys in a range [lo, hi], hash tables require O(N) scan. Use a BST or Segment Tree for O(log N) range queries.
Rule of thumb: If you only need O(1) exact key lookup, insert, and delete — use Hashing. If you need ordering, ranges, or sorted traversal — use a BST-based structure.

C (Custom Chaining)
// Simple chaining hash map in C
#define SIZE 7

typedef struct Node {
    int key, value;
    struct Node* next;
} Node;

Node* table[SIZE];

void insert(int key, int value) {
    int idx = key % SIZE;    // hash function
    Node* node = (Node*)malloc(sizeof(Node));
    node->key   = key;
    node->value = value;
    node->next  = table[idx]; // prepend to chain
    table[idx]  = node;
}

C (Custom Chaining)
int search(int key) {
    int idx = key % SIZE;
    Node* cur = table[idx];

    // Traverse the chain at this bucket
    while (cur != NULL) {
        if (cur->key == key)
            return cur->value;  // found!
        cur = cur->next;
    }
    return -1;  // not found
}
JavaScript

C (Custom Chaining)
void deleteKey(int key) {
    int idx = key % SIZE;
    Node* cur  = table[idx];
    Node* prev = NULL;

    while (cur) {
        if (cur->key == key) {
            if (prev) prev->next = cur->next;
            else      table[idx] = cur->next;
            free(cur);
            return;
        }
        prev = cur; cur = cur->next;
    }
}

💥 Collision Resolution

Critical Concept
1
Chaining (Separate Chaining)
▼

Each slot in the hash table holds a linked list (or dynamic array) of all keys that hashed to that index. On collision, the new entry is simply appended to the chain. This is the most common approach used by Python, Java HashMap, and C++ unordered_map.

  • Simple to implement and handles high load factors well
  • Extra memory for pointers; potential cache misses
  • Worst case O(N) if all keys hash to same slot
2
Open Addressing (Linear Probing)
▼

Instead of a linked list, all entries live directly in the table. On collision, probe for the next empty slot: idx = (h(key) + i) mod capacity. Better cache performance; works best with low load factors (< 0.7).

  • Linear Probing: check idx+1, idx+2 … (clustering issues)
  • Quadratic Probing: check idx+1², idx+2² … (less clustering)
  • Double Hashing: use a second hash function for step size

Real-World Analogies

Hashing shows up everywhere in real life. Think of it as finding a locker number from your student ID:

1
Library Card Catalog
You look up a book by title → the index points to the exact shelf location. No need to scan every book. O(1) lookup!
2
School Locker System
Your student ID (key) maps to a locker number (hash index). Two students might be assigned same locker → collision, assign next free locker.
3
Phone Contacts
You search "Alice" → instantly found. Behind the scenes your phone stores contacts in a hash map by name for O(1) retrieval.
4
Password Verification
Websites store hash(password) not the password itself. On login: hash(input) → compare with stored hash. Even if stolen, raw password stays safe.
