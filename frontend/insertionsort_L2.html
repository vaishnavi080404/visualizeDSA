Try Different Scenarios

Best Case — Already Sorted

When the array is already sorted, each key element only requires one comparison per iteration — compare with the immediate left neighbour, no shift needed. This results in O(N) time, a huge advantage over Merge Sort and Quick Sort for nearly-sorted inputs.

🗄 Array: [5, 10, 15, 20, 25, 30] — sorted, O(N) comparisons
📋 Operation Log

Average Case — Random Order

Random input — each key shifts past roughly half the sorted portion on average. Results in O(N²/4) comparisons and shifts overall. Watch the key travel leftwards, nudging elements to the right until its spot is found.

🗄 Array: [29, 10, 14, 37, 13, 8] — random order
📋 Operation Log

Worst Case — Reverse Sorted

Reverse-sorted input forces every key to shift past the entire sorted portion — maximum shifts per iteration. The 2nd element shifts 1, 3rd shifts 2, … Nth shifts N-1. Total: O(N²) comparisons and shifts.

🗄 Array: [60, 50, 40, 30, 20, 10] — reverse sorted, max shifts
