LRU Cache

Redis, CPU caches, browser cache — all use LRU eviction. A doubly linked list keeps items ordered by recency. get() moves to head; when full, tail (least used) gets evicted.

Time & Space Complexity

Linked lists shine at dynamic insertion/deletion — but pay an access cost. Know the tradeoffs cold.

Operation Singly LL Doubly LL Circular LL Notes
Insert at Head O(1) O(1) O(1) Redirect head pointer only
Insert at Tail O(n) O(1)* O(1)* *O(1) with tail pointer maintained
Delete at Head O(1) O(1) O(1) Move head to head.next
Delete at Middle O(n) O(n) O(n) Must find node via traversal first
Delete (with ptr) O(n) O(1) O(n) Doubly LL: use prev pointer directly
Access by Index O(n) O(n) O(n) No random access — must traverse
Search O(n) O(n) O(n) Linear scan from head
Space (total) O(n) O(n) O(n) Doubly uses extra ptr per node

Advantages & Disadvantages

Every data structure is a tradeoff. Know when linked lists win — and when they lose to arrays.

✅ Advantages
✓

O(1) insertion and deletion at head

No shifting like arrays. Prepending or removing the first node just redirects a pointer.

✓

Truly dynamic size

No pre-allocated capacity. Grows and shrinks exactly as needed — no overflow risk, no wasted slots.

✓

O(1) middle insertion with a pointer

If you already have a reference to a node, inserting before or after it is just pointer manipulation — no shifting needed.

✓

Foundation for other structures

Stacks, queues, deques, adjacency lists, hash chaining — all built on linked lists. Understand this, understand everything.

✓

Efficient merging and splitting

Merge two sorted linked lists in O(n) without extra memory. Split a list at any point in O(1) — just break a pointer.

❌ Disadvantages
✗

No random access — O(n) indexing

To reach the 10th element you must traverse nodes 1-9. Arrays give you arr[9] in O(1). This is a real pain point.

✗

Extra memory for pointers

Every node stores a pointer (8 bytes on 64-bit). A singly LL of integers uses 2x memory. Doubly uses 3x compared to an array.

✗

Poor cache performance

Array elements are contiguous in memory — the CPU can prefetch them. Linked list nodes are scattered — each access is a cache miss.

✗

No binary search

Even if the list is sorted, you can't binary search it — you can't jump to the middle in O(1). Must use skip lists for that.

✗

Reverse traversal (singly)

Singly linked lists can only go forward. Reversing requires either a doubly LL or rebuilding the whole list — O(n) either way.

🎯
When to Use Linked List — and When to Avoid It

This is the real interview question: "Why a linked list here?" Have a sharp answer.

✅ Use Linked List When...
➕

Frequent insertions / deletions at head or middle

Prepend-heavy workloads, task schedulers, message queues — arrays would shift millions of elements. LL does it in O(1).

📏

Size is unknown at compile time

You don't know how many elements you'll get. Linked lists grow on demand without reallocation or copying overhead.

🔗

Implementing other structures

Building a stack, queue, deque, or adjacency list? Linked list is your building block. push = insertHead, pop = deleteHead.

🔄

Circular / round-robin patterns

OS round-robin scheduling, music loops, carousel slides — circular LL gives infinite cycling with O(1) next operation.

↔️

Need bidirectional traversal

Browser history, undo/redo — doubly LL lets you walk forward and backward without rebuilding. prev + next pointers.

✂️

Merging / splitting sequences

Merge sort on linked lists, polynomial addition, symbol tables — pointer relinks beat any array approach for these.

❌ Avoid Linked List When...
🎲

You need random access

Use an Array or ArrayList. If your code says list.get(i) frequently, you're paying O(n) every time. Switch to array — it's O(1).

🔍

Frequent searching by value

Use a HashMap or TreeMap. Searching a linked list is always O(n) — no shortcuts, no binary search, no hash lookup.

💾

Memory is tight

Use a compact array. Every node carries pointer overhead (8+ bytes). An array of 1000 ints uses 4KB; a linked list uses 12–20KB for the same data.

⚡

Cache performance matters

Use arrays or array-backed structures. Scattered memory = constant cache misses. For read-heavy, performance-critical loops, arrays are 5-10x faster in practice.

📊

You need to sort the data

Use sort() on an array. Sorting a linked list is possible (merge sort) but awkward and has higher constant factors than an in-place array sort.

🔢

Small, fixed-size collections

Just use an array. The pointer overhead and cache miss cost completely dominate for small n. Static arrays are almost always faster for n < 100.
