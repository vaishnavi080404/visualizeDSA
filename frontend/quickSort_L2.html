📋 PSEUDO CODE
BASE CASE: 0 or 1 element
if low ≥ high: return
PIVOT: Last element
pivot = arr[high]
i POINTER: Boundary init
i = low − 1
SCAN j: Traverse left→right
for j = low to high−1
COMPARE: Is arr[j] ≤ pivot?
if arr[j] ≤ pivot
SWAP i↔j: Grow left region
i++ ; swap(arr[i], arr[j])
PLACE PIVOT: Final position
swap(arr[i+1], arr[high])
RECURSE: Left & right sub-arrays
quickSort(low, i)
quickSort(i+2, high)

BASE CASE: 0 or 1 element
if low ≥ high: return
PIVOT: First element
pivot = arr[low]
INIT POINTERS: i & j from ends
i = low−1 ; j = high+1
ADVANCE i: Find element ≥ pivot
do i++ while arr[i] < pivot
ADVANCE j: Find element ≤ pivot
do j-- while arr[j] > pivot
CHECK: Pointers crossed?
if i ≥ j: return j (boundary)
SWAP i↔j: Move elements
swap(arr[i], arr[j])
RECURSE: Two sub-arrays
quickSort(low, j)
quickSort(j+1, high)
SORTED: All partitions done ✓
array fully sorted

Try Different Scenarios

Best Case — Balanced Partitions

Pivot always lands near the middle of the sub-array. Each partition creates two roughly equal halves, so recursion depth stays at log₂(N), giving O(N log N).

🗄 Array: [3, 10, 20, 30, 40, 50, 60] — pivot lands near middle
📋 Operation Log

Average Case — Random Input

Random array — pivot lands somewhere in the middle on average, giving roughly balanced partitions. This is Quick Sort’s typical real-world performance at O(N log N).

🗄 Array: [10, 80, 30, 90, 40, 50, 70] — random order
📋 Operation Log

Worst Case — Already Sorted (Lomuto)

With sorted input and Lomuto’s last-element pivot, every pivot is the largest element — partitions into 0 + (N−1) sub-arrays. Recursion depth = N, giving O(N²).

🗄 Array: [10, 20, 30, 40, 50, 60] — sorted: worst for Lomuto
